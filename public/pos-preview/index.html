Task 4 · Employee Concession POS — Build Plan

Reviewed against the live Task 2 database. Tables: concessions, transactions, transaction_items, inventory_log, admin_users. The POS reuses all of them — no duplicate data.

Architecture

  • Lives at /pos (separate area, own session ALEX_POS_SESS) — independent of the public site and /admin.
  • Products read live from concessions where is_available=1. Deactivate in admin → vanishes here automatically.
  • One-tap add to cart (client-side cart); stock badges shown at add time but not locked.
  • Checkout endpoint api/pos-checkout.php: opens ONE DB transaction → SELECT … FOR UPDATE on every cart product → re-verifies stock atomically → writes transactions + transaction_items + decrements concessions.stock_quantity + logs inventory_log → commits. Any shortfall rolls back the whole order.
  • Records source_channel = staff_register so Task 3 reports separate walk-up from online automatically.

Auth & security

  • PIN login (4–6 digits), bcrypt-hashed. 5 failed attempts → 15-min lockout. Generic "incorrect PIN".
  • 8-hour shift session; Lock screen (re-enter PIN) without ending shift; full Log Out.
  • Rate-limited login + checkout endpoints; CSRF on every mutation (house pattern).

Concurrency (highest risk)

  • Two iPads ringing the last item: the FOR UPDATE row lock means exactly one wins; the other gets "Sorry — [Item] just sold out. Remove it or reduce quantity." No partial orders, no overselling.
